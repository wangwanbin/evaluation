# AI 能力评估系统 — 设计大纲

> **定位**：这是一个**评测夹具（Evaluation Harness）**——系统本身不运行模型、不提供推理服务。
> 被评测的模型由用户通过 **OpenAI 兼容 API** 接入（vLLM 本地部署 / 云端 API / 任意兼容端点）。
> 系统的职责：发 Prompt → 收 Response → 评分 → 出报告。
>
> **目标**：构建一个便携、自包含、模块化的 AI 能力评估平台，部署在移动硬盘中即插即用。
> **第一版覆盖**：大模型基础能力 · RAG 能力 · Agent 能力，后续按需扩展。
> **核心原则**：离线可用、数据可复现、结果可对比、模块可插拔。

## 〇、配置管理（.env）

**唯一必填项**：被评测模型的 vLLM 地址。其余全有默认值。

```bash
# .env（从 .env.example 复制，只改一行）
# ===== 必需 =====
EVAL_MODEL_API_BASE=http://192.168.1.100:8000/v1   # ← 只改这个！vLLM 地址
EVAL_MODEL_API_KEY=                                 # API Key（本地 vLLM 通常留空；云端 API 必填）

# ===== 以下都有默认值，不改也能跑 =====
EVAL_MODEL_NAME=qwen2.5-72b                        # 默认值，vLLM 部署的模型名
EVAL_CONCURRENCY=10                                # 并发数
EVAL_TIMEOUT=120                                   # 单题超时

# ===== Judge 模型（评测 RAG/Agent 答案质量时才需要） =====
JUDGE_MODEL_API_BASE=http://192.168.1.100:8000/v1  # 可以和被评测模型用同一个 vLLM
JUDGE_MODEL_API_KEY=                               # 同上
JUDGE_MODEL_NAME=qwen2.5-72b

# ===== RAG 系统端点（P1 阶段） =====
RAG_API_BASE=http://192.168.1.100:8080/ask
RAG_API_KEY=
```

| 原则 | 说明 |
|------|------|
| 最小配置 | 只改 `EVAL_MODEL_API_BASE` 即可开始 LLM 评测 |
| `.env` 不入 Git | 模板文件 `.env.example` 提供参考 |
| 无需 API Key | 本地 vLLM 默认无鉴权，留空即可 |

## 一、总体架构

### 1.0 Docker 全容器化便携方案（核心设计）

**场景**：从移动硬盘拷贝 `eval-system/` 到目标 Linux 机器 → 改一行 `.env` → 一条命令跑起来。

```
移动硬盘（拷贝到目标机器后）/
└── eval-system/              # 拷贝这个目录即可，其他什么都不需要
    ├── images/                # 🔑 预构建好的镜像（按版本号命名）
    │   ├── eval-app-v1.0.0.tar  #     文件名 = 版本号，永不覆盖
    │   └── eval-app-v1.1.0.tar  #     升级时新增文件，旧版本保留备回滚
    ├── docker/
    │   ├── docker-compose.yml #     直接引用预构建镜像
    │   ├── Dockerfile.app     #     保留 Dockerfile 仅备查，正常流程不构建
    │   └── Dockerfile.ui
    ├── core/                  #     核心框架
    ├── modules/               #     评估模块
    ├── datasets/              #     评测数据集（已打包，无需下载）
    ├── results/               #     评测结果（初始为空目录）
    ├── .env.example           #     配置模板
    └── start.sh               #     一条命令：加载镜像 → 启动
```

**用户操作流程（3 步，1 分钟）**：

```bash
# 1. 配置（只改一个字段）
cp .env.example .env
# 编辑 .env，只改这一行：
#   EVAL_MODEL_API_BASE=http://192.168.x.x:8000/v1   ← vLLM 的地址

# 2. 一键启动
./start.sh

# 3. 开始评测
docker exec -it eval-app eval run llm --bench mmlu
```

**`start.sh` 做了什么**：

```bash
#!/bin/bash
# 全程不需要 docker build，不需要 pip install，不需要下载任何东西

# 从 docker-compose.yml 读取需要的版本号
VERSION=$(grep 'image: eval-app:' docker/docker-compose.yml | sed 's/.*://')

# 1. 环境预检（5 秒）
check_docker          # 有 docker 就行，20.10+
check_storage_driver  # 必须 overlay2
check_disk            # 剩余空间 ≥ 5GB
check_port 8000       # 端口不冲突

# 2. 加载镜像（30 秒，只做一次）
#    检查本地是否有对应版本的镜像，没有则从版本号命名的 .tar 加载
if ! docker image inspect eval-app:$VERSION >/dev/null 2>&1; then
    TAR_FILE="images/eval-app-${VERSION}.tar"
    if [ -f "$TAR_FILE" ]; then
        docker load -i "$TAR_FILE"
    else
        echo "❌ 缺少镜像文件: $TAR_FILE，请联系开发者获取此版本。"
        exit 1
    fi
fi

# 3. 启动服务
docker compose -f docker/docker-compose.yml up -d

# 4. 验证
sleep 3
curl -s http://localhost:8000/health && echo "✅ eval-system ready"
```

**docker-compose.yml**（引用预构建镜像，不 build）：

```yaml
services:
  app:
    image: eval-app:v1.0.0        # 版本标签，与 images/*.tar 中的版本号一致
    # build: ...                  # 不构建！镜像已预打包
    env_file: .env
    volumes:
      - ./datasets:/app/datasets
      - ./results:/app/results
    ports: ["8000:8000"]
```

> `docker-compose.yml` 中的 `image` 标签和 `images/` 文件夹中的 `.tar` 文件名通过版本号一一对应。
> 升级时只改 `image:` 标签和新增 `.tar` 文件，不覆盖旧文件。

### 1.1 便携性设计要点

| 要点 | 方案 |
|------|------|
| 环境隔离 | **Docker 全容器化**，目标机器只需 Docker，不装 Python、不装 pip、不装任何依赖 |
| 预构建镜像 | `images/*.tar` 随项目分发，`start.sh` 自动 `docker load`，**不需要 docker build** |
| 一键启动 | `./start.sh`：预检 → 加载镜像 → 启动 → 验证，全程 1 分钟内完成 |
| 数据集内置 | 常用评测集（MMLU、GSM8K 等）直接打包在 `datasets/` 目录中 |
| 极简配置 | **唯一必填**：`EVAL_MODEL_API_BASE`（vLLM 地址）。其他全有默认值 |
| 零外网依赖 | 不拉镜像、不下载数据集、不访问 CDN。vLLM 通常也在同一台机器上 |
| 跨平台 | Docker 屏蔽 OS 差异，Linux / macOS / Windows 均可运行 |
| 报告便携 | 结果输出在 `results/` 目录，Markdown 报告可直接查看 |

### 1.1.1 镜像更新流程

版本号驱动，不覆盖旧文件：

```bash
# === 开发者侧 ===

# 1. 改代码，更新版本号
echo "v1.1.0" > VERSION

# 2. 构建新镜像
docker build -t eval-app:v1.1.0 -f docker/Dockerfile.app .

# 3. 导出（文件名含版本号）
docker save -o images/eval-app-v1.1.0.tar eval-app:v1.1.0

# 4. 更新 docker-compose.yml 中的版本标签
#    将 image: eval-app:v1.0.0 改为 image: eval-app:v1.1.0

# 5. 复制到移动硬盘（旧版本 .tar 保留不删，方便回滚）
cp images/eval-app-v1.1.0.tar /mnt/eval-system/images/
cp docker/docker-compose.yml   /mnt/eval-system/docker/

# === 用户侧 ===

# 6. 拷贝新的 eval-system/ → 启动
./start.sh
# → 读到 image: eval-app:v1.1.0 → 检查本地无此标签 → 加载 v1.1.0.tar
```

**回滚**：把 `docker-compose.yml` 中 `image:` 改回旧版本号，重新 `./start.sh` 即可。

### 1.2 核心设计模式：插件化模块注册

```python
# 每个评估模块只需实现统一接口，即可被核心框架自动发现和调度

class EvalModule(ABC):
    name: str           # 模块唯一标识，如 "llm", "rag", "agent"
    version: str        # 模块版本号
    description: str    # 模块描述

    def list_benchmarks(self) -> list[Benchmark]: ...
    def run_benchmark(self, benchmark_id: str,
                      model_config: ModelConfig,
                      params: dict) -> EvalResult: ...
    def get_metrics(self) -> list[MetricDef]: ...
```

新模块只需在 `modules/` 下创建一个子目录，继承 `EvalModule` 即可自动注册。

### 1.3 模型接入规范

```
│   · 发 Prompt         │    /v1/chat/completions  │   · vLLM (推荐)        │

│   · 评分              │                         │   · 云端 API           │
│   · 出报告            │                         │   · 任意兼容端点        │
└──────────────────────┘                         └──────────────────────┘
```

## 二、模块一：大模型基础能力评估（LLM）

### 2.1 评估维度总览

```
LLM 能力评估
├── 1. 知识储备          → MMLU, C-Eval, CMMLU
├── 2. 推理能力
│   ├── 数学推理          → GSM8K, MATH
│   ├── 逻辑推理          → LogiQA, BIG-Bench
│   └── 常识推理          → HellaSwag, PIQA, WinoGrande
├── 3. 代码能力          → HumanEval, MBPP, LiveCodeBench
├── 4. 指令遵循          → IFEval, MT-Bench
├── 5. 长文本理解        → LongBench, Needle-in-a-Haystack
├── 6. 多语言能力        → FLORES, XQuAD, MGSM
├── 7. 安全性/对齐        → TruthfulQA, ToxiGen, RealToxicityPrompts
└── 8. 效率指标          → 推理延迟、吞吐量、显存占用、成本
```

### 2.2 支持的数据集适配方案

| 数据集 | 类型 | 获取方式 | 离线方案 |
|--------|------|---------|---------|
| MMLU | 多选 | HuggingFace / 内置 | 预下载 JSON 打包 |
| C-Eval / CMMLU | 中文多选 | HuggingFace | 预下载打包 |
| GSM8K | 数学 | HuggingFace | 内置 8K 题 |
| MATH | 数学 | HuggingFace | 预下载打包 |
| HumanEval | 代码 | GitHub | 内置 164 题 |
| MBPP | 代码 | HuggingFace | 预下载打包 |
| HellaSwag / PIQA | 常识 | HuggingFace | 内置 |
| IFEval | 指令遵循 | GitHub | 内置 JSON |
| MT-Bench | 对话质量 | lmsys | GPT-4 评判（需 API）或内置参考分 |
| Needle-in-a-Haystack | 长文本 | GitHub | 动态生成长文本，不依赖外部数据 |
| TruthfulQA | 安全 | HuggingFace | 预下载打包 |

### 2.3 评测执行流程

```
输入：模型配置（API endpoint / 本地路径）+ 评测项选择
  ↓
Step 1: 加载数据集 & 构建 Prompt（支持自定义 system prompt）
  ↓
Step 2: 调用模型推理（支持并发 / 限速）
  ↓
Step 3: 答案提取 & 评分（规则匹配 + LLM-as-Judge）
  ↓
Step 4: 指标汇总 → EvalResult
  ↓
输出：单评测项得分 + 多维度雷达图 + 错误样本
```

### 2.4 关键设计决策

- **答案提取**：优先规则匹配（正则/解析），复杂任务用 LLM-as-Judge
- **LLM-as-Judge**：支持配置 Judge 模型（GPT-4 / Claude / 本地模型），Judge 链可复现
- **Few-shot 支持**：内置各数据集的标准 few-shot 模板，也支持自定义
- **模型接入**：只通过 OpenAI 兼容 API 接入。`.env` 中配置 `EVAL_MODEL_API_BASE`，系统发 HTTP 请求，不加载模型、不管理权重、不启动推理服务

## 三、模块二：RAG 能力评估

> **定位**：用户的 RAG 系统是黑盒。评测系统不提供向量库、Embedding 模型、文档分块——这些用户已有。
> 评测只做一件事：发送预设问题到 RAG API → 收到答案 → 对比标准答案 → 打分。

### 3.1 评测模型

```
┌──────────────┐    POST /ask     ┌──────────────────────┐
│  评测系统      │ ──────────────► │   用户的 RAG 系统       │
│              │                 │   (向量库+检索+LLM)     │
│  预设问题集    │ ◄────────────── │                       │
│  + 标准答案    │    JSON 应答     └──────────────────────┘
└──────────────┘
        │
        ▼
  Judge LLM 对比打分
  → Answer Correctness
  → 延迟
```

### 3.2 .env 配置

```bash
# RAG 系统端点（用户自己部署的 RAG 服务）
RAG_API_BASE=http://192.168.1.100:8080/ask    # RAG 问答接口
RAG_API_KEY=                                   # API Key（本地部署通常留空）
RAG_API_FORMAT=openai                          # 请求格式：openai / custom
```

### 3.3 评测数据格式

```json
{
  "id": "rag_001",
  "question": "2024 年成都市常住人口是多少？",
  "answer": "成都市 2024 年末常住人口为 2153.5 万人。",
  "metadata": {
    "difficulty": "easy",
    "category": "社会经济"
  }
}
```

### 3.4 评测流程

```
加载评测集 → 逐条 POST 到 RAG_API_BASE → 收集应答 → Judge 打分 → 报告
                                                         ↑
                                              JUDGE_MODEL (独立模型)
```

### 3.5 评测指标与引用溯源

| 指标 | 含义 | 前置条件 |
|------|------|---------|
| Answer Correctness | RAG 答案 vs 标准答案 | 标准答案 |
| Citation Accuracy | 答案中的引用是否指向正确来源 | **RAG 系统返回引用** |
| Latency | 端到端耗时 | 无 |

**🔑 引用溯源是评分的前提**：

```
评测系统发送请求时，要求 RAG 系统返回格式：
{
  "answer": "成都市 2024 年末常住人口为 2153.5 万人。",
  "citations": [                                    ← 必须包含！
    {"chunk_id": "chengdu_stats.md#3", "text": "..."}
  ]
}

如果 RAG 系统只返回 { "answer": "..." } 不带 citations：
  → 系统报错 ❌
  → "RAG 系统未返回引用信息（citations），无法进行溯源评分。
     请在 RAG 系统中开启引用返回功能，或使用 --no-citations 跳过引用评测。"
  → 评测中止（不产出不可靠的评分）
```

**`--no-citations` 降级模式**：

```bash
# 跳过引用评测，只做 Answer Correctness + Latency
eval run rag --bench my_domain --no-citations
```

此时评分报告会标注 `⚠️ 引用评测已跳过，Citation Accuracy 不计入总分`。

### 3.6 数据来源

| 路径 | 说明 |
|------|------|
| **内置** | 预打包的 QA 对（如 MS MARCO 子集转为问答格式） |
| **手工** | `datasets/rag/custom/` 下按 JSON 格式提供 |
| **导入** | `eval rag import <file>` 支持 JSON/CSV |

### CLI

```bash
eval run rag --bench ms_marco    # 用内置评测集
eval run rag --bench my_domain   # 用手工制备的评测集
```

## 四、模块三：Agent 能力评估（P2 规划）

> **定位**：用户的 Agent 系统是黑盒。评测系统发送任务 → 收到执行结果 → 对比预期 → 打分。
> MVP 阶段不实现，但架构预留接口。

### 4.1 评测模型（同 LLM/RAG 一致）

```
┌──────────────┐   POST /execute  ┌──────────────────────┐
│  评测系统      │ ──────────────► │   用户的 Agent 系统     │
│              │                 │   (工具链+规划+执行)    │
│  任务集        │ ◄────────────── │                       │
│  + 预期结果    │    JSON 结果     └──────────────────────┘
└──────────────┘
        │
        ▼
  Judge LLM 判定
  → 任务完成率
  → 执行耗时
```

### 4.2 .env 配置（预留）

```bash
AGENT_API_BASE=http://192.168.1.100:9000/execute   # Agent 执行端点
```

### 4.3 评测数据格式（预留）

```json
{
  "id": "agent_001",
  "task": "查询订单 #12345 的物流状态，如果已签收则发确认邮件",
  "expected_outcome": "邮件已发送，内容包含 '已签收'",
  "tools_required": ["order_lookup", "email_send"],
  "max_steps": 5
}
```

### 4.4 评测指标（预留）

| 指标 | 含义 |
|------|------|
| Task Success Rate | 任务是否达到预期结果 |
| Steps Efficiency | 实际步骤数 vs 预期最少步骤数 |
| Latency | 端到端执行耗时 |

> Agent 沙箱环境（Docker 隔离执行 + 资源熔断 + 分级网络）在 P2 期按需实现，MVP 不做。

### 5.2 结果存储与版本管理

**存储方案：SQLite（WAL 模式）+ JSON 文件双轨**

| 数据 | 存储方式 | 用途 |
|------|---------|------|
| 评测元数据（run 记录、模型配置、指标汇总） | SQLite | 查询、对比、排行榜 |
| 评测详情（逐题结果、Agent 轨迹） | JSON 文件（`results/runs/`） | 完整原始数据，人类可读 |
| 报告 | HTML/PDF/Markdown（`results/reports/`） | 独立分发 |

**SQLite 安全配置（防损坏）**：

```python
# 应用启动时自动执行
PRAGMA journal_mode=WAL;          # 🔑 WAL 模式：写不阻塞读，崩溃后可自动恢复
PRAGMA synchronous=NORMAL;        # 安全与性能平衡（FULL 过慢，OFF 危险）
PRAGMA foreign_keys=ON;           # 外键约束
PRAGMA busy_timeout=5000;         # 写锁等待 5 秒，避免 BUSY 错误
PRAGMA cache_size=-64000;         # 64MB 缓存，减少磁盘 I/O
```

**自动备份机制**：

```bash
# 每次评测完成后自动触发（非阻塞）
# 备份策略：保留最近 5 次的备份，超过自动清理
results/
├── eval.db                    # 主数据库
├── backups/
│   ├── eval_20260610_095500.db   # 最新
│   ├── eval_20260610_083000.db
│   ├── eval_20260609_180000.db
│   ├── eval_20260609_120000.db
│   └── eval_20260609_090000.db   # 最旧（自动删除）
```

**结果目录结构**：

```
results/runs/{timestamp}_{benchmark}/
├── report.html          # 📊 HTML 报告（主格式，浏览器打开）
├── report.md            # 📄 Markdown 报告（终端预览）
├── data.json            # 📦 结构化原始数据
└── logs/                # 运行日志
```

**崩溃恢复流程**：

```
1. 启动时检测 WAL 文件是否存在（异常退出标志）
2. 若存在，自动执行 PRAGMA wal_checkpoint(TRUNCATE) 回放
3. 若数据库损坏，自动从最新备份恢复：
   ├── 尝试 PRAGMA integrity_check
   ├── 若失败 → 重命名损坏文件 → 复制最新备份
   ├── 从 WAL + JSON 文件重建丢失的评测记录
   └── 输出恢复报告
```

**单写入点原则（防数据竞争）**：

```
❌ 错误架构：
  agent-sandbox 容器 → 直接写 /app/results/eval.db  ← 多进程同时写 = 文件锁冲突 + 损坏风险

✅ 正确架构：
  agent-sandbox 容器 → IPC / HTTP API → app 容器 → SQLite（唯一写入者）
                       ↑                    ↑
                   gRPC / Redis /      所有 DB 操作
                  Unix Socket         集中在此
```

| 规则 | 说明 |
|------|------|
| **app 是唯一写入者** | 所有 SQLite 写操作必须通过 app 服务的 API |
| **agent-sandbox 只读** | Agent 沙箱可以读取 JSON 结果文件，但禁止直接操作 `.db` |
| **通信方式** | 容器间通过 HTTP API（`app:8000`）或 Redis 队列传递评测结果 |
| **优势** | 消除文件锁竞争、崩溃时只有一侧需要恢复、架构清晰 |

> **为什么不用 PostgreSQL / MySQL？** 保留了 SQLite 的零配置优势（移动硬盘即插即用），同时通过 WAL + 自动备份消除了损坏风险。对于评测系统这种写多读少的场景，WAL 模式下 SQLite 性能完全足够（单次评测写入 ~1000 行，秒级完成）。

**结果对比**：
- `eval compare <run_id_1> <run_id_2>` — 两次评测的指标对比
- `eval leaderboard <benchmark>` — 历史排行榜

## 六、报告与可视化

### 6.1 报告内容

每份评测生成 HTML（主格式）和 Markdown 双份报告，包含：

```
1. 概览面板
   ├── 模型基本信息（名称、端点）
   ├── 评测时间 & 耗时
   └── 总得分概览

2. 多维度雷达图
   ├── LLM：知识 / 推理 / 代码 / 指令遵循 / 多语言 / 安全
   ├── RAG：检索质量 / 生成忠实度 / 端到端 / 延迟
   └── Agent：工具使用 / 规划 / 环境交互 / 记忆 / 安全

3. 逐项明细
   ├── 每个 benchmark 的得分 & 对比基线
   ├── 错误样本展示（输入 / 模型输出 / 正确答案）
   └── 置信区间

4. 对比分析（多次评测）
   ├── 折线图（指标随时间推进）
   ├── 胜率矩阵（模型 A vs 模型 B）
   └── 交叉分析（如 chunk_size 对 RAG 指标的影响）

   ├── 薄弱环节高亮
```

> 每次评测完成后自动生成 HTML + Markdown 双份报告。HTML 文件用浏览器直接打开即可。

## 七、命令行接口（CLI）

**支持多种等价写法**，同一个命令可以写成不同形式，解析结果完全一致：

```bash
# 以下四条命令完全等价
eval run llm --bench mmlu --concurrency 10
eval run llm -b mmlu -c 10
eval run llm --bench=MMLU --concurrency=10
eval llm run mmlu -c 10                        # 位置参数（bench 作为第一个位置参数）
```

### 参数写法规则

| 写法 | 示例 | 说明 |
|------|------|------|
| 长选项 | `--bench mmlu` | 空格分隔 |
| 等号 | `--bench=MMLU` | 等号分隔，值不区分大小写 |
| 短选项 | `-b mmlu` | 单字母缩写 |
| 位置参数 | `eval llm run mmlu` | benchmark 作为第一个位置参数 |
| 逗号分隔 | `-b mmlu,gsm8k` | 多个值用逗号连接 |
| 重复标志 | `-b mmlu -b gsm8k` | 多次指定同一选项 |

## 十一、MVP 优先执行策略

> **核心原则：先窄后宽，先跑通再扩展。**

### MVP 要做的事（两周内可交付）

```
          ┌──────────────────┐
          │   eval 最小系统    │
          ├──────────────────┤
          │ ✅ LLM 评测       │  MMLU + GSM8K，两个 benchmark
          │    · 通过 .env    │  被评测模型→OpenAI API
          │     配置 API     │  Judge 模型→独立 API
          │    · 并发请求     │
          │    · 规则评分     │
          ├──────────────────┤
          │ ✅ RAG 检索评测   │  MS MARCO 子集
          │    · 预制备数据   │  Precision/Recall/NDCG/MRR
          │    · 不生成数据   │
          ├──────────────────┤
          │ ✅ CLI 全流程     │  eval run → eval report
          ├──────────────────┤
          │ ✅ Markdown 报告  │  基础报告 + 得分表
          └──────────────────┘
```

### MVP 明确不做的事

| 不做的事 | 原因 | 预计何时做 |
|---------|------|-----------|
| Agent 沙箱 | 复杂度高，先验证评测框架骨架 | P1（第二期） |
| HTML 交互报告 + 雷达图 | 等全流程稳定后再美化 | P1（第二期） |
| Web UI | CLI 够用，先验证核心逻辑 | P2（后期） |
| RAG 评测集自动生成 | 数据可信度问题，用预制备数据更可靠 | P2（后期） |
| 全量 benchmark | 2 个跑通后加更多边际成本很低 | P1 逐步加入 |

### 验证标准（MVP 通过条件）

- [ ] 拷贝 `eval-system/` 到任意 Linux 机器，`./start.sh` 一键启动（1 分钟内）
- [ ] 只改 `.env` 中 `EVAL_MODEL_API_BASE` 一行，`eval run llm --bench mmlu` 即可跑完 57 个类别
- [ ] `eval run rag --bench ms_marco` 跑完检索评测并输出 Precision/Recall
- [ ] Judge 模型和被评测模型可以是**同一个 vLLM 实例**，配置独立不混用
- [ ] 断电后重启，SQLite 自动恢复，不丢数据
- [ ] 评测完成后自动生成 `report.html`（浏览器打开）+ `report.md`（终端预览）双份报告
- [ ] 全程不需要外网：不拉镜像、不下载数据集、不访问 CDN

### 为什么先做 MVP

1. **尽快暴露问题**：便携性、Docker 兼容性、路径问题等，越早遇到越容易修
2. **尽快拿到成果**：两周出一个可演示版本，比三个月出一个"功能全但有坑"的版本好得多
3. **边际成本递减**：LLM 评测框架跑通后，加新 benchmark 只是加数据集+评分函数，工作量很小
4. **RAG/Agent 需要 LLM 框架做基础**：没有稳定的模型调用和评分管道，RAG 和 Agent 评测无从谈起
